- [Single entry pipeline — one human-invoked CLI, four-layer architecture, recipe-based recovery](single_entry_pipeline.md) — Factory has exactly one human-invoked CLI during normal operation: `npx tsx tools/run.ts <spec-id> [<spec-id>...]`. The runtime decomposes into four layers — driver / phases / lifecycle / recovery — with CLI-as-agent-protocol on the lifecycle boundary (the CLI invocation IS the structured event; agents don't return JSON). Multi-spec runs are sequential and dependency-aware first; parallel later is designed-for, not committed-to. Recovery is scenario-keyed recipes (8 scenarios; auto-recoverable for `ProviderTransient`, `ProviderUnavailable`, `BuildFailed`, `StaleBranch`, `AgentNonResponsive`; escalate-only for `LintFailed`, `TestFailed`, `CompletionGateBlocked`) with two-layer provider failover (cross-CLI ordered `persona_providers` list + within-CLI `model_failover` for abstraction providers like copilot). `LintFailed`/`TestFailed` always escalate by design — auto-recovery would invite agents to disable rules or mutilate tests. Budget: 1 attempt per scenario per packet per phase, 3 total per packet. **locked 2026-04-30** (recovery model refined same day). Informed by [factory_script_audit](../research/factory_script_audit.md); implementation lessons recorded in [ajv_migration_attempt](../research/ajv_migration_attempt.md) (Phase 4.6 revert) and [phase_6_recovery_attempt](../research/phase_6_recovery_attempt.md) (Phase 6 revert).
- [Spec artifact model — human-authored markdown one layer above intents, 1:1 spec → intent parity](spec_artifact_model.md) — A new artifact type — spec — sits one layer above the existing intent. Specs are human-authored markdown at `specs/<spec-id>.md` with minimal frontmatter (`id`, `title`, optional `depends_on`); the body is the authoritative content the planner reads. Each spec maps 1:1 to an intent; factory translates spec → intent at run time. The intent remains the locked downstream-consumed schema (with one additive backward-compatible `depends_on` field added); the spec is the source of truth, the intent is derived state regeneratable from disk. The post-Phase-8 convergence pass introduced a runtime governance gate split: spec-driven runs bypass the gate (human approval happened when the spec was written); intent-driven runs against hand-authored `intents/<id>.json` must carry `status: approved`/`planned`/`delivered`. **locked 2026-04-30**; approval-gate split documented 2026-05-10. Informed by [factory_script_audit](../research/factory_script_audit.md).
- [Memory scope split — stateless workers, stateful project; host-project memory vs factory-development memory](memory_scope_split.md) — Memory operates at two scopes. Workers (per-packet agent invocations) stay stateless — short-lived, parallelizable, replaceable, recovery via typed scenarios rather than learned context (claw-code's posture). The project layer is stateful — accumulating runs, curated and persistent (claurst's three-layer pattern). Within project scope, factory respects a host/guest boundary claurst doesn't have to navigate, splitting into host-project memory (factory writes to the host's tracked artifact tree; the host owns the loading mechanism) and factory-development memory (the memdir-style symlink convention this repo uses; per-developer setup, NOT inherited by host projects). Memory categories at host-project scope when implemented: architectural facts, recurring failures, code patterns. Operator preferences are out of scope. The write-side and consolidation pass are deferred to a future spec; the architectural shape that implementation will take is committed in [host_project_memory_graph_rag](host_project_memory_graph_rag.md). **locked 2026-04-30** (clarification of host vs factory-development sub-scopes landed same day). Informed by [claurst_audit](../research/claurst_audit.md) §9 and [claw_code_audit](../research/claw_code_audit.md) §12.
- [Host-project memory — graph-based RAG with typed nodes, composite weights, semantic+graph retrieval, continuous consolidation](host_project_memory_graph_rag.md) — When factory implements the host-project memory write-side (deferred per `memory_scope_split.md`), it will be a typed knowledge graph with semantic retrieval, weighted nodes, and continuous consolidation — NOT a flat catalog of markdown notes. Nodes are typed (`architectural-fact`, `pattern`, `failure-mode`, `component`, `convention`, `provenance`); edges are typed (`depends-on`, `supersedes`, `contradicts`, `instance-of`, etc.); each node carries a composite weight vector (reinforcement decay, authority, graph centrality, recency, contradiction flag, retired) combined at retrieval time. Retrieval is a single pipeline: embedding similarity + k-hop graph traversal + composite ranking + MMR + per-agent token budget. Source of truth is human-readable markdown in `factory/memory/<category>/<id>.md`; a derived SQLite + embedded vector store at `factory/memory/.index/` is gitignored. Extraction is best-effort at pipeline end; consolidation (incremental + periodic deeper pass) is first-class. A flat-catalog approach is sufficient for human-curated decisions but insufficient for agent-consumed institutional memory because it lacks retrieval ranking, node centrality, contradiction detection, and reinforcement/decay learning. Reference architectures: Microsoft GraphRAG, Mem0, LightRAG, Cognee, MemGPT. **locked 2026-05-11**. Specific A-decisions (embedding model, vector store dependency, retrieval-injection scope, consolidation cadence, contradiction-resolution policy) are deferred to 4-6 future implementation specs across 3 staging steps.
- [Factory event observability — typed events with provenance labels as the observability backbone](event_observability.md) — Factory emits typed events at every meaningful state transition during pipeline execution. The taxonomy is a closed TypeScript union (unknown types rejected); each event carries a provenance label (`live_run`, `test`, `healthcheck`, `replay`, `dry_run`) so consumers can filter appropriately. Events are written append-only to the host's tracked artifact tree as a structured `{ event_type, timestamp, provenance, payload }` stream — free-form `console.log` is not an event. The recovery layer matches recipes on event types rather than on raw stderr; the future memory write-side will extract facts from `live_run`-labelled streams without test pollution; future operator tooling (doctor, dashboards, an external dispatcher) consumes the same stream. Events are an observability stream, not the state of record — packet status remains authoritative in `packets/<id>.json`. **locked 2026-05-01**; informed by [claw_code_audit](../research/claw_code_audit.md) §4 (lane events) and §6 (recovery consumes events).
- [Factory cost visibility — per-invocation tokens and dollars, configurable caps at run/packet/per-day scope](cost_visibility.md) — Factory tracks the cost of every agent invocation (input/output tokens plus a derived dollar figure where the provider supports it) and surfaces aggregate cost per packet, per spec, and per pipeline run. Configurable caps at run, packet, and per-day scope abort or escalate rather than silently continue. Cost records are written to the host's tracked artifact tree alongside completions; the host owns the audit trail. Captured at the agent-call boundary in `tools/pipeline/agent_invoke.ts`. Foundational for budget-aware downstream features — recovery retry budgets, multi-spec scheduling, and the manager-executor tiering pattern lifted from claurst — all of which assume per-call cost data exists. **locked 2026-05-01**; informed by [claurst_audit](../research/claurst_audit.md) §4.4 (`BudgetSplitPolicy`) and §10 (cost tracking).
- [Recovery uses recipes, not a policy DSL — scenario-keyed TypeScript functions over composable And/Or conditions](recovery_recipes_not_dsl.md) — Phase 6's recovery layer uses scenario-keyed recipes — a data-driven map from `FailureScenario` to `RecoveryRecipe` typed `(scenario, context) => RecoveryAttempt | EscalateRequest` — rather than claw-code's `PolicyEngine`-style DSL with composable `And`/`Or` conditions over a `LaneContext`. Recipes are TypeScript, not config, and live in `tools/pipeline/recovery.ts` as a constant map. Eight scenarios, eight recipes; adding one means adding an enum variant and a recipe entry. The DSL is overkill for factory's failure-mode catalog and would impose a small DSL on host-project operators without proportional return. claw-code's DSL exists because lanes are concurrent and policies cross lanes; factory's pipeline is sequential through Phase 5+, so cross-lane composition isn't a problem we have today. If recipes ever sprout `if` chains, a condition-evaluation library can be layered on top mechanically. **locked 2026-05-01**. Design choice was reaffirmed during the [phase_6_recovery_attempt](../research/phase_6_recovery_attempt.md) revert — recipes were correct, state-machine integration needed redesign.
- [Reviewer CLI exception — review.ts is the verdict channel, kept honest by ReviewDecisionMissing](reviewer_cli_exception.md) — The code-reviewer agent invokes `review.ts --approve` or `review.ts --request-changes` as the protocol channel for its verdict, in deliberate contrast to the planner, developer, rework, and QA prompts, which are explicitly instructed NOT to call lifecycle CLIs. The reviewer is the exception because its output IS a structured verdict the pipeline must consume reliably to decide whether to advance to `complete` or loop back to `rework`, and there is no mechanical way to derive that verdict from an exit code — it lives in the model's judgement about the code. The exception is kept honest by the `ReviewDecisionMissing` recovery scenario: when the reviewer exits 0 without invoking `review.ts`, the develop-phase closure synthesizes a failure carrying `REVIEW_DECISION_MISSING_MARKER`, the classifier matches by exact substring, and the recipe escalates with budget 0 — no silent force-approve. The prompt-exception and the failure-path are coupled by design; without `ReviewDecisionMissing` the exception would be a facade. **locked 2026-05-11** during the post-Phase-8 convergence pass.
- [Verification grading deferred — binary build/lint/test through Phase 8, no Green Contract tiers](verification_grading_deferred.md) — Factory does not adopt claw-code's "Green Contract" graded verification (the `TargetedTests`/`Package`/`Workspace`/`MergeReady` ordered tier model) in `single-entry-pipeline.md` or its sister specs through Phase 8. Verification stays binary: build, lint, and test each pass or fail; a packet is complete only when all three pass. The current model is over-rigorous (every packet runs every check), not under-rigorous — the cost is wasted CI time, not shipped bugs. The right tier vocabulary depends on host-project tooling shape and factory doesn't yet have enough host-project diversity to design that well. Adding it later is additive (optional `required_verification_level` field on packet schema). Revisit post-Phase 8 against accumulated host-project experience, likely composed with cost visibility. **locked 2026-05-01**; informed by [claw_code_audit](../research/claw_code_audit.md) §7.
- [Doctor diagnostic deferred — no unified preflight command until after Phase 8](doctor_diagnostic_deferred.md) — Factory does not add claw-code's `claw doctor` unified-preflight equivalent in `single-entry-pipeline.md` or its sister specs through Phase 8. Health-checking remains fragmented across `tools/status.ts` (factory state), `tools/validate.ts` (schema and integrity), manual provider checks, and the pre-commit completion-gate hook. A unified `factory doctor` command is high-value for operator UX but not on the architectural critical path — its lack costs setup friction, not correctness. Better designed late, once Phases 5-8 surface the full set of checks worth running (multi-spec sequencing, recovery state, cost-cap status, event stream health, worktree state). Revisit as a sister spec post-Phase 8. **locked 2026-05-01**; informed by [claw_code_audit](../research/claw_code_audit.md) §10.
- [Factory self-governance — factory is a tool other projects use, it does not govern its own development](factory_self_governance.md) — Factory is a tool other projects use; it is not a self-hosting product. The pipeline runner (`run.ts`) is not invoked against this repo. Specs in `specs/` are implementation roadmaps for human + Claude collaborative execution, not inputs to autonomous factory runs — they're read by humans and Claude during design and implementation sessions; they don't decompose into intents → packets → autonomous execution the way host-project specs do. The canonical `factory.config.json` lives at `templates/factory.config.json` (the host-project template); the root `factory.config.json` is a non-canonical development convenience because `tools/config.ts` and pre-commit hooks expect a loadable config in the working directory. The deferral of self-governance is deliberate, not a bootstrapping concession waiting for stabilization. **locked 2026-04-30**, re-establishing principle going forward after an earlier (now-discarded) 2026-03-22 memory note flagged the same posture during bootstrapping.

## Research and lessons (reference material, lives in `../research/`)

- [claurst research audit — Rust reimplementation of Claude Code; manager-executor, memdir, single-loop patterns](../research/claurst_audit.md) — Code-level audit of `claurst` (GPL-3.0, ~13 MB, 205 Rust files across 12 crates), surveyed 2026-04-30. Started as a clean-room reimplementation of Claude Code from a `spec/` directory and evolved into a TUI pair programmer that now treats Claude Code as one provider behavior among many. Informs three load-bearing factory decisions: cost visibility (`Arc<CostTracker>` shared across nested agent calls, `BudgetSplitPolicy` enum — §4.4, §10), memory scope split (three-layer memdir + session_memory + AutoDream subsystem treating memory as load-bearing — §9), and single-entry pipeline (the `run_query_loop` single-loop constraint where sub-agents, slash commands, and the manager are all the same loop with different configs — §3, §4). Other patterns worth tracking: typed sub-crate decomposition by concern (12 crates) rather than by pipeline phase, worktree-per-spec isolation, manager-executor model where a larger manager (Opus/Pro) delegates to smaller executors (Sonnet/Haiku) through the existing `AgentTool` infrastructure.
- [claw-code research audit — autonomous Discord-driven coding harness; lane events, recovery recipes, doctor preflight](../research/claw_code_audit.md) — Code-level audit of `claw-code` (~23 MB; Rust workspace + Python reference; 10 Rust crates), surveyed 2026-04-30. Autonomous harness whose primary human interface is Discord — humans set direction by sentence, agent "claws" coordinate work in parallel across planning/execution/review/recovery without human intervention. End-state worker is offline while the system continues, which gives it an unusually mature failure model. Informs five factory decisions: event observability (lane events as a typed closed enum with provenance labels — §4), recovery recipes vs DSL (claw-code's `PolicyEngine` with composable `And`/`Or`, which factory chose NOT to adopt because lanes-are-concurrent isn't a problem factory has — §8), verification grading deferred (Green Contract `TargetedTests`/`Package`/`Workspace`/`MergeReady` tiers — §7), doctor diagnostic deferred (unified `claw doctor` preflight — §10), and the memory scope split (claw-code's deliberate omission of learned memory; every worker starts fresh — §12). claw-code's autonomy posture is more aggressive than factory's "human approves intent, agents execute"; the audit makes that contrast load-bearing rather than incidental.
- [Factory script audit — honest evaluation of tools/ on idempotency and usefulness, ahead of the pipeline redesign](../research/factory_script_audit.md) — Self-audit of factory at v0.2.0 (post-pipeline-redesign in commit `5a6df82`), surveyed 2026-04-30. Honest evaluation of all 12 TypeScript files in `tools/` (3,544 lines total) plus `setup.sh`, `setup.ps1`, and `hooks/pre-commit`, scored on idempotency and usefulness. The headline finding — `run.ts` at 798 lines with zero tests doing seven distinct jobs, pipeline-idempotent in the orchestrator sense but the load-bearing weakness in the script surface — drove the four-layer decomposition committed in `single_entry_pipeline.md`. The "strict scripts + idempotent orchestrator" coupling identified here is what the post-decision idempotency rewrite of all four lifecycle scripts was designed to dissolve. The audit is the canonical input to the architecture move that followed.
- [The ajv migration that wasn't — Phase 4.6 bounded-iteration revert lesson](../research/ajv_migration_attempt.md) — Bounded-iteration revert lesson recorded 2026-05-01. Phase 4.6 was originally scoped as a pure-refactor migration of four hand-rolled artifact validators in `tools/validate.ts` (785 lines, the largest file in `tools/`) to ajv consuming the existing `schemas/*.schema.json` files. After three rounds of codex GPT-5.5 review, the verdict was RECOMMEND-REVERT: ajv enforced full-schema strictness the old validators ignored (`additionalProperties: false`, optional field constraints, `oneOf` mutex semantics); successive lax-mode filters and drop-rule mini-DSLs fixed regressions but failed to reduce complexity (785 → 1,396 lines across three files plus a new dependency plus 46 regression-pinning tests). What shipped instead (revised Phase 4.6, commit `192e971`): integrity-layer extraction to `tools/pipeline/integrity.ts` (467 lines), `validate.ts` shrunk 785 → 423 lines, 29 compatibility tests pinning existing hand-rolled behavior, no ajv, no new dependencies. The lesson: when migrating a hand-rolled validator to a schema-driven engine, first verify the hand-rolled validator was actually doing schema-driven validation; if it's doing semantic checks dressed up as schema validation, the migration adds layers without removing complexity. First case where the bounded-iteration model surfaced an architectural error in the brief itself, not in the implementation. Original branch `worktree-agent-aa2a30d634d859b5c` was force-deleted post-revert.
- [The Phase 6 recovery attempt that wasn't — bounded-iteration revert lesson on cross-cutting layer integration](../research/phase_6_recovery_attempt.md) — Bounded-iteration revert lesson recorded 2026-05-05. Phase 6 was scoped as the recovery layer per `recovery_recipes_not_dsl.md` — 8 failure scenarios with scenario-keyed recipes, per-scenario per-packet retry budgets, 4 new event types, wiring at every `invokeAgent` site and `complete.ts` boundary, plus a pure/I-O/orchestration split. Three rounds of codex review: R1 recipe action labels propagated but actions never executed (CLAUDE.md §3.5 facade — tests pinned labels not behavior); R2 actions wired but `StaleBranch` was production-unreachable and QA `BuildFailed` remediation passed the wrong packet to `buildDevPrompt`; R3 RECOMMEND-REVERT — `StaleBranch` escalation at `request_review` was observable (events fired, escalation files written) but not controlling — the failed `runWithRecovery` result was only logged, then the state machine advanced to review unconditionally. CLAUDE.md §3.3 violation: failure must be both visible AND controlling. The lesson: when adding an orthogonal cross-cutting layer (recovery, tracing, auth) to an existing state machine, the integration IS the architecture, not the wiring; the cross-cutting layer's signals must be both observable AND controlling, and tests must drive the calling state machine to confirm both. Second case (after `ajv_migration_attempt`) where the bounded-iteration model surfaced an architectural error in the brief. The pure recovery module, classifier, recipes, retry budgets, event types, `runGitRebase`, and BuildFailed dev-agent remediation are all salvageable for the revised attempt; the state-machine integration is what needs to be designed first-class, likely via a typed return discriminator from `runWithRecovery`. Original branch `phase-6-recovery` was force-deleted post-revert.
- [Recurring workflow patterns — recognition signals from session work that should govern future sessions](../research/recurring_workflow_patterns.md) — Meta-lesson recorded 2026-05-12 generalizing from the two concrete revert lessons above plus the session experience that produced Phases 1-8, the post-5.7 orchestrator decomposition, the convergence pass, the backlog cleanup, DEP0190, the host-project memory graph-RAG decision, and the Tier 1 memory alignment. Eight named recognition signals an orchestrator or reviewer should watch for during architectural work, each with session examples + the corrective move when the signal fires. Patterns: (1) bounded-iteration model — three review rounds maximum, then RECOMMEND-REVERT + write a research note; (2) test-pins-the-bug — when an `it(...)` description doesn't match its assertions, the test is pinning the bug under a misleading name; (3) doc-creates-inconsistency — documenting a mechanism as load-bearing surfaces silent fallbacks that were previously invisible (the doc didn't create the bug, it made the inconsistency visible); (4) observable-vs-controlling — cross-cutting signals must be both visible AND dispatched on, per CLAUDE.md §3.3; (5) state-machine-integration-is-architecture — the integration of an orthogonal layer into an existing state machine is the load-bearing design problem, not the wiring; (6) reverted-lesson preservation — every revert produces a `type: lesson` research note before the branch is force-deleted; (7) codex hang-and-restart — wrap `codex exec` dispatches in a SIGKILL watchdog because the process hangs at startup in 5/8 dispatches; (8) FI-7 distinct identity — orchestrator / developer / QA / reviewer are four separate identities (Opus / Opus fresh / Opus fresh / codex GPT-5.5), never collapsed. The doc is auto-loaded into every factory-development session; new patterns are appended when a failure shape has surfaced twice and been named both times. **locked 2026-05-12**.
